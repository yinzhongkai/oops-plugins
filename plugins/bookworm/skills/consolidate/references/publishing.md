# 成稿规范

consolidate 的 4.4 成稿步骤产出可发布文稿。本文件定义文稿的结构、质量、与“拐杖”的边界。

## 1. 成稿的前提

成稿是走完 4.1 共忆 → 4.2 费曼 → 4.3 关联后的奖赏。**未走完前三步不得成稿**。这是内部门控，由 skill 在成稿前校验。

## 2. 内容来源铁律

文稿的内容必须来自**用户自己在共忆/费曼环节讲出来的话**。Claude 的职责仅限：

- **结构组织**：把用户散述的内容归到合理章节结构。
- **文字润色**：通顺语句、统一术语、修正语病，不改变意思。
- **补充连接**：段间过渡、承上启下。

Claude **不得独立从原文合成文稿**。若用户在某点讲不出（4.1 时没回忆起来），该点不得由 Claude 凭原文补全进文稿——应回到共忆环节让用户补，或在文稿中标注“此处回忆不全”。

这是堵死“第二根拐杖”的关键：成稿若变成 Claude 照原文重写，用户拿到的就是 Claude 的理解而非自己的理解，违背“能用自己的话讲清楚”的目标。

## 3. 文稿结构

```markdown
# 《书名》第X章：<本章主题>

> 一句话概括本章核心。

## 核心观点
<2-4 个核心论点，每个一句陈述 + 用户的展开>

## 论证脉络
<作者如何论证这些观点，用用户的话复述>

## 我的理解
<用户自己的延伸、联想、与现实的对照>

## 与前后章的关联
<本章在全书论证链中的位置，承接了什么、引出了什么>

## 待深入
<回忆不全或仍存疑的点，诚实标注>
```

## 4. 质量标准

- **用用户的话**：保留用户的表达习惯和例证，不强求书面化到失去个人色彩。
- **诚实标注**：回忆不全的部分如实标“待深入”，不假装完整。
- **可发布**：通顺、连贯、结构清晰，达到可分享给他人阅读的程度。
- **篇幅适度**：覆盖本章核心即可，不必逐点铺陈。

## 5. 成稿后

- 文稿存到 `<书名>/notes/chXX.consolidate.md`。
- 在 progress.md “环节记录”表追加：⑤ 巩固 章 X 完成（已成稿）。
- 向用户展示文稿，并提示：可继续下一章（回到 pre-read），或回头补“待深入”部分。
