# 三阶段流水线定义

本文件是 inkstone 插件的唯一流水线定义来源。所有 skill 引用本文件以保持一致，避免在多处重复定义导致漂移。

## 1. 流水线总览

把素材改造成平台文案分三个阶段，按顺序推进。每个阶段有对应的 skill。

| 阶段 | skill | 触发 | 产物 | 时机 |
|------|-------|------|------|------|
| ① 前采集 | gather | 斜杠 + 自动 | 选题目录建立 + 原始素材就位 | 每个选题一次 |
| ② 中整理 | organize | 斜杠 + 自动 | `<选题>/source.md`（统一中间格式） | 每个选题一次 |
| ③ 后适配 | adapt | 斜杠 + 自动 | `<选题>/<平台>.md`（平台文案） | 每个平台一次 |

另有 `methodology` skill（自动激活）作为路由层，判断用户处于哪阶段并引导到对应 skill，本身不干活。

## 2. 顺序约束

- ① 必须最先：没有采集就没有素材，后续整理无从谈起。
- ② 在 ① 之后、③ 之前：所有来源先归一成统一中间格式，平台适配才能无差别消费。
- ③ 在 ② 之后：必须基于确认过的 source.md 适配，不在原始素材上直接写文案。

允许跳阶段直达：用户指定素材和目标平台时，methodology 自动按序补齐前置阶段（见 methodology 第 5 节），不需用户手动分步。

## 3. 阶段职责边界

- **gather（前）只采集不评判**：把素材原样抓下来、提取关键信息、决定文案归属目录。不判断素材好不好、不改写素材。
- **organize（中）只整理不创作**：把异构素材归一成 source.md 的结构化字段。不针对任何平台、不写平台文案。
- **adapt（后）只适配不改义**：基于 source.md 按平台风格规范产出文案。不回到原始素材重新采集、不改变 source.md 已确认的核心观点。

## 4. 目录约定

阅读笔记（输入）与文案（输出）分目录存放，与 bookworm 共用同一套项目级约定：

```
<project>/
├── books/                              # bookworm 产出的阅读笔记（inkstone 的输入来源）
│   └── <书名>/
│       ├── notes/
│       │   ├── chXX.consolidate.md     # 感悟文档（gather 主要素材）
│       │   └── chXX.pre-read.md        # 章节预读（可作关联参考）
│       └── framework.md                # 全书框架（可作关联参考）
└── copywriting/                        # inkstone 产出的平台文案
    ├── <书名>/                         # 来源于某本书
    │   └── <选题>/
    │       ├── raw-source.md           # ① gather 采集留底（organize 输入）
    │       ├── source.md               # ② 中间格式
    │       ├── wechat.md               # ③ 公众号文案
    │       ├── zhihu.md                # ③ 知乎文案
    │       └── xiaohongshu.md          # ③ 小红书文案
    └── <其他>/                         # 来源于链接/话题等
        └── <选题>/
            ├── raw-source.md
            ├── source.md
            └── ...平台文案
```

### 4.1 文案归属判断（gather 职责）

gather 不主动扫描感悟文档，由用户显式传素材路径 `/inkstone:gather <素材> [选题名]`，按路径前缀判断归属：

- 素材路径在 `books/<书名>/` 下 → 文案落 `copywriting/<书名>/`
- 其余位置（链接、话题、项目外文件）→ 文案落 `copywriting/<其他>/`

### 4.2 选题目录名

- 书来源：用书名 slug（如 `intelligent-investor`）。
- 非书来源：用户显式传选题名则用之；否则从链接标题/话题内容自动生成 slug；都没有则用 `topic-<日期>` 兜底（日期由用户提供或交互确认，不在 skill 内取系统时间）。

## 5. 中间格式契约（source.md）

source.md 是 organize 产出、adapt 消费的唯一契约。所有来源必须归一成此结构，平台适配无差别读取。格式：

```markdown
---
选题: <选题名>
来源类型: book | url | topic
来源路径或链接: <感悟文档路径 / 网址 / 话题描述>
整理时间: <YYYY-MM-DD>
---

## 核心观点
（一句话讲清这篇要表达什么）

## 关键论据素材
（支撑观点的论据、数据、例子）

## 可引用金句
（适合直接用的原句、感悟、用户原话）

## 受众
（写给谁看：年龄/职业/兴趣/痛点）

## 备注
（风格倾向、禁忌、平台倾向建议等）
```

字段说明：

- **frontmatter** 记录元数据，供 methodology 判断进度、adapt 确认来源。
- **核心观点** 是文案的主轴，adapt 所有平台都围绕它展开，不得偏离。
- **关键论据素材** 是血肉，adapt 按平台取舍详略。
- **可引用金句** 是抓手，尤其小红书钩子、公众号点睛。
- **受众** 决定语气和深浅。
- **备注** 留给用户标注特殊要求。

### 5.1 字段缺失处理

- 核心观点缺失 → 不进 adapt，回 organize 补。
- 其余字段缺失 → 在 source.md 标注「待补」，adapt 可继续但向用户提示该字段缺失可能影响成品质量。

## 6. 三条铁律

1. **改造而非创作**：输入是已有素材，输出是适配调性的文案。adapt 不凭空生成 source.md 里没有的核心观点；素材不足时回 gather/organize 补，不在 adapt 里编。
2. **organize 后必经用户确认**：source.md 整理完，必须展示给用户确认/编辑后才能进 adapt。中间格式是质量分水岭，把确认权交给用户。methodology 自动串链路时也在此停下等用户。
3. **前后端对称扩展**：加来源只在 gather 内加 `references/<来源>.md` 并登记一行；加平台只在 adapt 内加 `references/<平台>.md` 并登记一行。两端的扩展互不干扰、互不重写对方。

## 7. 中文语感基准

所有文案默认**自然口语化的中文**，贴近真人创作：

- 避免明显的 AI 腔（如「首先…其次…最后」的机械排比、空洞的「在当今社会」、过度对仗）。
- 保留口语节奏和真实表达，不强求书面化到失去个人色彩。
- 各平台在此基础上再调风格（详见 adapt 的 `references/<平台>.md`）。

## 8. 环境要求

- 无额外运行时依赖。感悟文档由 Claude 原生读取能力处理，链接抓取用内置 `WebFetch`，话题搜索用内置 `WebSearch`。
- 链接遇付费墙 / 反爬 / 登录墙时，gather 报错并提示用户手动粘贴正文，不静默失败。
