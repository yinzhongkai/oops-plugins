# 感悟文档采集手法

gather 采集 bookworm 产出的感悟文档（consolidate 成稿）作为素材。本文件定义读取结构、提取要点、关联参考的做法。

## 1. 感悟文档是什么

bookworm 的 `/bookworm:consolidate` 产出每章的「可发布文稿」，存于：

```
books/<书名>/notes/chXX.consolidate.md
```

它是结构化 markdown，内容来自读者自己的话（Claude 只做结构组织 + 文字润色），保留读者表达习惯。这是 inkstone 最主要的素材来源——读者读透一章后自己的理解，天然适合改造成社媒文案。

## 2. 感悟文档结构

典型结构（来自 bookworm 成稿规范）：

```markdown
# 《书名》第X章：<本章主题>

> 一句话概括本章核心

## 核心观点
<2-4 个核心论点，每个一句陈述 + 读者的展开>

## 论证脉络
<作者如何论证这些观点，用读者的话复述>

## 我的理解
<读者自己的延伸、联想、与现实的对照>

## 与前后章的关联
<本章在全书论证链中的位置>

## 待深入
<回忆不全或仍存疑的点>
```

## 3. 提取要点

读取感悟文档后，按 source.md 中间格式的字段方向提取（提取 ≠ 整理成 source.md，那是 organize 的事；这里是把原文关键信息抓出来留底）：

| source.md 字段 | 从感悟文档哪里取 |
|----------------|------------------|
| 核心观点 | 「核心观点」章节 + 文首一句话概括 |
| 关键论据素材 | 「论证脉络」章节 |
| 可引用金句 | 「我的理解」中读者自己的精炼表达、文首概括 |
| 受众 | 感悟文档通常不含，留空待 organize 补 |
| 备注 | 「待深入」章节（标注哪些点读者自己也没想透，adapt 时需谨慎或回避） |

「待深入」章节重要：它诚实标注了读者没读透的点。采集时务必带入留底，organize 整理时转写进 source.md 的备注，提醒 adapt 不要在这些点上强行展开。

## 4. 关联参考（可选）

同一本书还有两类文件可作关联参考，提升素材厚度：

- **`books/<书名>/framework.md`**：全书框架。把握本章在全书的位置，避免 adapt 写出的文案脱离全书语境。
- **`books/<书名>/notes/chXX.pre-read.md`**：章节预读骨架。含章节结构和引导问题，可佐证核心观点。

默认只采感悟文档本身；用户要求「结合全书」或素材偏薄时，再读这两类文件补充。不要默认全读，避免上下文膨胀。

## 5. 多章合并

用户可能一次给多个感悟文档路径（如「把这三章的感悟合成一篇公众号」）。此时：

- 逐个读取、各自提取要点。
- 留底到一个 `raw-source.md`，按章分节标注来源章号。
- 选题名由用户指定或从主题归纳（多章通常有共同主题）。
- 归属判断：多个感悟文档同属一本书 → 落 `copywriting/<书名>/`；跨书则向用户确认归属。

## 6. 留底格式

把提取要点写入 `<选题>/raw-source.md`，结构对齐感悟文档原文，便于 organize 参考：

```markdown
---
来源类型: book
来源路径: books/<书名>/notes/chXX.consolidate.md
---

# 原始素材留底

## 核心观点
<从感悟文档提取>

## 论证脉络
<从感悟文档提取>

## 我的理解
<从感悟文档提取>

## 待深入
<从感悟文档提取，标注待 adapt 谨慎的点>

（若关联了 framework.md / pre-read.md，附关键摘录于此）
```

## 7. 健壮性

- **路径读不到**：确认路径正确、文件存在；用户给的是目录而非文件时，列出该目录下 `notes/*.consolidate.md` 供用户选。
- **文件不是 consolidate 产物**（结构不符）：向用户确认是否仍要采集；是则按通用 markdown 读取，不强套章节结构。
- **「待深入」整章都是**（读者没读透）：如实报告，提示用户该章素材质量有限，adapt 时会有局限——是否继续由用户定。
